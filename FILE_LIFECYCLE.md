# מנגנון לייפסייקל קבצים | File Lifecycle

מסמך זה מגדיר איך הצוות מנהל קבצים פעילים בפרויקט: גרסאות, ארכיון, יומן שינויים, וכללי מחיקה. **כל סוכן שכותב או משנה קובץ פעיל מחויב לעקוב אחר הפרוטוקול הזה.**

---

## 1. למה צריך את זה?

עד עכשיו, כל שינוי קובץ היה דריסה ישירה - הגרסה הקודמת נעלמה. זה יוצר 4 סיכונים:

1. **אין דרך לחזור אחורה.** אם ריי הריצה ראיון מחדש והעדפה החדשה פחות טובה מהקודמת - הקודמת איננה.
2. **אין היסטוריה.** אי אפשר לראות איך העסק התפתח לאורך זמן, או מה השתנה ומתי.
3. **אין הגנה מטעויות.** עריכה ידנית שגויה אחת = אובדן נתונים.
4. **אין שקיפות.** המשתמש לא יודע מה נעשה אצלו ומתי.

המנגנון הזה פותר את ארבעתם.

---

## 2. אילו קבצים נמצאים בניהול לייפסייקל?

| קובץ | מי כותב | תכיפות שינוי | קריטיות |
|------|---------|----------------|---------|
| `BUSINESS_PROFILE.md` | ריי (וגם המשתמש ידנית) | נמוכה-בינונית | **קריטית** - כל הצוות תלוי בו |
| תוצרים שמורים (מאמרים, לוחות תוכן, וכו') | המשתמש בוחר | משתנה | נמוכה-בינונית |

אם בעתיד יתווסף סוכן שכותב קובץ פעיל - הוא מתחייב לאותו פרוטוקול.

---

## 3. מבנה התיקיות

```
[project root]/
├── BUSINESS_PROFILE.md              ← הגרסה הפעילה (תמיד אחת)
├── CHANGELOG.md                      ← יומן שינויים מצטבר
└── archive/
    ├── README.md                     ← הסבר על תיקיית הארכיון
    ├── BUSINESS_PROFILE.2026-05-15.md   ← גרסה ישנה (תאריך = מתי הוחלפה)
    ├── BUSINESS_PROFILE.2026-06-01.md
    └── ...
```

**הערה:** בשימוש דרך Claude Cowork, המבנה הזה נבנה בתוך הפרויקט של המשתמש. הפלאגין עצמו מכיל רק תבניות (CHANGELOG ריק, תיקיית archive ריקה עם README).

---

## 4. פרוטוקול ההחלפה (חובה לכל סוכן שכותב קובץ פעיל)

לפני שמחליפים קובץ פעיל בגרסה חדשה, הסוכן מבצע את 4 הצעדים בסדר הזה:

### צעד 1: בדיקת קיום
האם קיימת גרסה קודמת של הקובץ?
- **לא:** דלג ישר לצעד 4 (כתיבה ראשונה - אין מה לארכב).
- **כן:** המשך לצעד 2.

### צעד 2: ארכוב הגרסה הקודמת
העתק את הקובץ הקיים ל-`archive/` עם תאריך התחלת ההחלפה:

```
פורמט: archive/[FILENAME].{YYYY-MM-DD}.md
דוגמה: archive/BUSINESS_PROFILE.2026-05-27.md
```

**אם כבר קיים קובץ בארכיון מאותו תאריך** (החלפה שנייה באותו יום) - הוסף סיומת קצרה: `.2026-05-27.b.md`, `.2026-05-27.c.md` וכו'.

### צעד 3: כתיבת הגרסה החדשה
החלף את הקובץ הפעיל בתוכן החדש.

### צעד 4: רישום ב-CHANGELOG
הוסף ערך חדש ל-`CHANGELOG.md` בראש הקובץ (הכי חדש למעלה):

```markdown
## 2026-05-27

- **עודכן:** BUSINESS_PROFILE.md
- **על ידי:** ריי (סבב ראיון מחדש לפי בקשת הבעלים)
- **סיבה:** הבעלים שינה את פוקוס העסק מ-X ל-Y, ביקש להריץ הקמה מחדש
- **גרסה קודמת:** archive/BUSINESS_PROFILE.2026-05-27.md
```

אם השינוי היה קל יחסית (תיקון של שדה אחד) - שורה אחת מספיקה:

```markdown
## 2026-05-27
- **עודכן ידנית:** BUSINESS_PROFILE.md - שדה "טווחי מחיר" עודכן
- **גרסה קודמת:** archive/BUSINESS_PROFILE.2026-05-27.md
```

## 5. כללי הארכיון

- **קבצים בארכיון הם read-only מבחינת הצוות.** סוכנים לא עורכים, לא מוחקים, ולא משנים אותם.
- **המשתמש יכול למחוק ידנית** ערכי ארכיון ישנים (מעל 90 יום) אם רוצה לחסוך מקום.
- **אם מתעורר ספק - לארכב, לא למחוק.**
- אם המשתמש מבקש מסוכן למחוק קובץ פעיל - הסוכן **מעביר אותו לארכיון** במקום למחוק, ומסביר שהקובץ זמין שם אם יחזור ויצטרך.

## 6. כללי המחיקה

| פעולה | מותר? |
|--------|-------|
| מחיקת קובץ פעיל בלי גיבוי | ❌ אסור |
| שכתוב קובץ פעיל בלי ארכוב הקודם | ❌ אסור |
| מחיקה אוטומטית של קבצי ארכיון | ❌ אסור |
| מחיקת קובץ ארכיון על ידי המשתמש ידנית | ✅ מותר (אחריותו) |
| העברת קובץ פעיל לארכיון לפי בקשת המשתמש | ✅ מותר |
| מחיקת קובץ פעיל אחרי שהוא מועבר לארכיון | ✅ מותר אם המשתמש אישר |

---

## 7. מה אם הקובץ נכשל באמצע ההחלפה?

לדוגמה: ריי התחילה לכתוב BUSINESS_PROFILE חדש, אבל הכתיבה נכשלה באמצע.

**פרוטוקול ההתאוששות:**

1. אם הארכוב כבר קרה אבל הכתיבה החדשה נכשלה: יש לנו רק את הגרסה בארכיון, וחסר קובץ פעיל. **שחזר את הגרסה הקודמת מהארכיון** ל-`BUSINESS_PROFILE.md`, וסמן ב-CHANGELOG שההחלפה לא הצליחה.
2. אם הארכוב לא הספיק לקרות: הגרסה הקודמת עדיין במקום. אין מה לעשות - פשוט נסה שוב.

**שום מצב לא צריך להישאר עם שום קובץ פעיל בלי גיבוי.**

## 8. דוגמה מלאה (ריי מריצה הקמה מחדש)

**מצב התחלתי:**
```

CHANGELOG.md                (עם 2 ערכים קודמים)
archive/
└── README.md
```

**ריי מתחילה ראיון חדש ב-2026-05-27.**

**שלב 1 - בדיקת קיום:** ✅ הקובץ קיים.

**שלב 2 - ארכוב:**
```
מקור:  BUSINESS_PROFILE.md
יעד:   archive/BUSINESS_PROFILE.2026-05-27.md
```

**שלב 3 - כתיבה:** ריי כותבת את הגרסה החדשה ל-`BUSINESS_PROFILE.md`.

**שלב 4 - רישום ב-CHANGELOG:**
```markdown
## 2026-05-27

- **עודכן:** BUSINESS_PROFILE.md
- **על ידי:** ריי (סבב ראיון מחדש)
- **סיבה:** הבעלים שינה את פוקוס העסק והעדיף להריץ הקמה מאפס
- **גרסה קודמת:** archive/BUSINESS_PROFILE.2026-05-27.md
```

```
└── BUSINESS_PROFILE.2026-05-27.md   (הגרסה הקודמת)
```

## 9. עיקרון הזהב

> **שינוי בלי גיבוי = איבוד נתונים שעוד יחזור לגרום נזק.**

הלייפסייקל הזה לא נראה דחוף עד שצריך אותו. וכשצריך - שמחים שהוא קיים. כל סוכן בצוות מחויב אליו.
